## Phase 0：技术验证（第 1-2 周）

**目标：跑通核心技术链路，确认可行性**

### 任务列表

| 周 | 任务 | 交付物 |
|---|------|--------|
| 1 | 搭建 BEATs 特征提取环境 | 可用编码器 |
| 1 | 下载并处理开源数据集（CatMeows, Dog vocalization） | 已标注数据集 |
| 1 | 训练 MLP 情境分类器 prototype | CV 准确率基线 |
| 2 | 搭建 LLM bridge（Claude API / GPT API） | 翻译结果 Demo |
| 2 | 录制 50 条自家猫狗叫声，端到端测试 | 测试报告 |
| 2 | 确认 MVP 技术选型 | 技术方案定稿 |

### 核心技术决策点

- **BEATs vs HuBERT vs WavLM** → 在测试集上对比表现
- **MLP 层数/宽度** → 从 1 层 128 单元开始，过拟合则加正则化
- **LLM 供应商** → 对比 Claude / GPT / 通义千问的成本和质量
- **云服务商** → 阿里云 vs AWS 国内节点 vs 腾讯云

### 风险控制

- 如果分类准确率 < 65%，考虑增加 NatureLM-audio 作为辅助信号
- 如果 LLM 响应时间 > 3 秒，考虑流式输出或模型蒸馏

## Phase 1：MVP 上线（第 3-8 周）

**目标：上线可用产品，获取种子用户**

### 第 3-4 周：后端 + AI 服务开发

| 任务 | 工时 | 依赖 |
|------|------|------|
| FastAPI 项目脚手架 | 2天 | - |
| 用户系统（注册/登录/JWT） | 3天 | - |
| 宠物档案 CRUD API | 2天 | - |
| 翻译 API（音频→特征→分类→LLM） | 5天 | Phase 0 模型 |
| 反馈 API | 2天 | 翻译 API |
| 数据库 Schema + 迁移 | 2天 | - |

### 第 5-6 周：客户端开发

| 任务 | 工时 | 依赖 |
|------|------|------|
| 录音模块（权限 + 波形 UI + 控制） | 3天 | - |
| 翻译结果展示页（卡片 + 置信度 + 推理） | 3天 | 翻译 API |
| 宠物管理页面 | 2天 | 宠物 API |
| 历史记录页面 | 2天 | 翻译 API |
| 反馈交互（点赞/点踩） | 1天 | 反馈 API |
| 用户引导 + 新手教程 | 2天 | - |

### 第 7 周：联调 + 内测

| 任务 | 说明 |
|------|------|
| 端到端联调 | 客户端→后端→AI→数据库→客户端 |
| 性能优化 | 目标 API 延迟 < 3s（含 AI 推理） |
| 错误处理 | 网络错误/空数据/超时等边界情况 |
| 内测招募 | 50-100 名种子用户（社群/朋友圈） |
| 内测反馈收集 | 问题清单 + 优先级排序 |

### 第 8 周：公测 + 发布

| 任务 | 说明 |
|------|------|
| App Store / 各应用商店提审 | 提前准备 |
| 修复 blocker bug | 内测暴露的问题 |
| 上线 | 🎉 |
| 监控告警配置 | Sentry + 服务器监控 |
| 发布内容 | 小红书/知乎/即刻/宠物品类社区 |

### MVP 核心指标

| 指标 | 目标（第 4 周末） |
|------|-----------------|
| 注册用户 | 1000+ |
| DAU | 200+ |
| 日均翻译次数 | 500+ |
| 翻译成功率 | > 95% |
| 平均响应时间 | < 3s |
| 统计显著的用户留存率（次日） | > 30% |
| 统计显著的反馈率 | > 10% 用户给了反馈 |

## Phase 2：增强功能（第 9-16 周）

**目标：核心差异化功能上线，建立竞争壁垒**

| LoRA adapter 训练管道 | 收集用户前 20 条反馈后自动训练 |

| A/B 测试框架 | 个性化 vs 通用模型效果对比 |

| 周报推送 | 每周推送宠物情绪总结 |

|------|------|
